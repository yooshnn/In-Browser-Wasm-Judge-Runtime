## 기능적 요구사항

### 제출 및 실행

- 시스템은 C++ 소스 코드를 입력받을 수 있어야 한다.
- 시스템은 stdio형 문제 사양을 입력받을 수 있어야 한다.
- 시스템은 문제 사양에 포함된 테스트케이스별 stdin을 기준으로 제출 코드를 실행할 수 있어야 한다.
- 시스템은 조립 없이 단일 제출 소스를 컴파일할 수 있어야 한다.
- 시스템은 컴파일 성공 시 테스트케이스별 실행을 수행할 수 있어야 한다.
- 시스템은 테스트케이스별 stdout, stderr, exit code를 수집할 수 있어야 한다.
- 시스템은 테스트케이스별 실행 시간을 측정할 수 있어야 한다.
- 시스템은 가능한 경우 테스트케이스별 memory bytes를 수집할 수 있어야 한다.

### 채점 방식

- 시스템은 exact judge를 지원해야 한다.
- 시스템은 custom JavaScript checker 기반 special judge를 지원해야 한다.
- exact judge에서는 기본 JavaScript checker로 stdout을 판정할 수 있어야 한다.
- custom judge에서는 `checkerId`를 통해 등록된 checker를 선택할 수 있어야 한다.
- 시스템은 실행 성공 결과를 checker에 전달하고 판정 결과를 반환할 수 있어야 한다.
- 시스템은 실행 실패를 checker에 위임하지 않고 직접 `JudgeTestResult`로 변환할 수 있어야 한다.

### 문제 사양

- 시스템은 문제별 식별자를 가질 수 있어야 한다.
- 시스템은 문제별 테스트케이스 목록을 정의할 수 있어야 한다.
- 시스템은 문제별 time limit를 정의할 수 있어야 한다.
- 시스템은 문제별 memory limit를 정의할 수 있어야 한다.
- 시스템은 문제별 checker 구성을 정의할 수 있어야 한다.
- exact checker를 사용하는 테스트케이스는 `expected: string`을 항상 가질 수 있어야 한다.

### 컴파일 구성

- 시스템은 compile 옵션을 raw flags 배열로 받을 수 있어야 한다.
- 시스템은 `CompilerPort` 호출 시 language와 flags를 함께 전달할 수 있어야 한다.
- 시스템은 특정 C++ 버전 enum에 API를 고정하지 않아야 한다.

### 결과 모델

- 시스템은 컴파일 실패를 구분해 반환할 수 있어야 한다.
- 시스템은 런타임 에러를 구분해 반환할 수 있어야 한다.
- 시스템은 시간 제한 초과를 구분해 반환할 수 있어야 한다.
- 시스템은 메모리 제한 초과를 구분해 반환할 수 있어야 한다.
- 시스템은 출력 제한 초과를 구분해 반환할 수 있어야 한다.
- 시스템은 accepted, wrong_answer 등의 최종 상태를 반환할 수 있어야 한다.
- 시스템은 전체 채점 소요 시간을 반환할 수 있어야 한다.
- 시스템은 테스트케이스별 실행 시간을 반환할 수 있어야 한다.
- 시스템은 가장 오래 걸린 테스트케이스의 실행 시간을 반환할 수 있어야 한다.
- 시스템은 테스트 단위 상태와 전체 채점 상태를 구분할 수 있어야 한다.

### 운영 인터페이스

- 시스템은 런타임 준비 상태를 조회할 수 있어야 한다.
- 시스템은 외부 애플리케이션이 사용할 수 있는 public API를 제공해야 한다.
- 시스템은 브라우저/Node 환경별 구현을 교체 가능해야 한다.

## 비기능 요구사항

### 실행 안정성

- 시스템은 사용자 코드를 메인 스레드가 아닌 별도 worker에서 실행해야 한다.
- 시스템은 실행 시간이 제한을 초과하면 해당 실행을 종료할 수 있어야 한다.
- 시스템은 출력 크기(stdout/stderr)를 제한할 수 있어야 한다.
- 시스템은 출력 크기(stdout/stderr) 제한을 초과한 실행을 실패로 처리할 수 있어야 한다.
- 시스템은 비정상 실행이 발생해도 전체 애플리케이션이 아닌 해당 실행 컨텍스트만 종료할 수 있어야 한다.

### 보안 및 격리

- 시스템은 사용자 코드가 DOM에 직접 접근하지 못하도록 해야 한다.
- 시스템은 사용자 코드가 네트워크에 직접 접근하지 못하도록 해야 한다.
- 시스템은 사용자 코드가 호스트 파일 시스템에 직접 접근하지 못하도록 해야 한다.
- 시스템은 checker가 OJ 목적에 필요한 최소한의 인터페이스만 사용하도록 해야 한다.

### 성능 및 배포

- 시스템은 브라우저 환경에서 동작해야 한다.
- 시스템은 대용량 wasm/runtime artifacts를 로딩할 수 있어야 한다.
- 시스템은 artifacts를 패키지 내부 또는 외부 배포 자산으로 분리할 수 있어야 한다.
- 시스템은 Node 기반 테스트 실행 환경으로 교체 가능해야 한다.
- 시스템은 artifact 참조와 preset을 별도 패키지로 분리할 수 있어야 한다.

### 유지보수성

- 시스템은 core가 browser/node 구현체에 직접 의존하지 않도록 설계되어야 한다.
- 시스템은 포트/어댑터 구조로 런타임 구현을 교체 가능해야 한다.
- 시스템은 stdio 채점 API를 기준으로 일관된 타입 인터페이스를 제공해야 한다.
- 시스템은 실행 엔진과 checker 인터페이스를 분리해야 한다.
- 시스템은 함수 직접 전달 대신 직렬화 가능한 checker 참조 모델을 우선해야 한다.
- 시스템은 `core` 패키지에서 DOM 타입과 runtime dependency 없이 유지되어야 한다.

## 핵심 유스케이스

### UC-01. 런타임 준비 상태 확인

사용자는 런타임이 준비되었는지 확인한다. 시스템은 compiler artifact, sysroot, executor 준비 여부와 capability를 반환한다.

### UC-02. exact checker로 stdio 제출 채점

사용자는 C++ stdio 코드를 제출한다. 시스템은 테스트케이스별 stdin으로 실행한 뒤 기본 JavaScript checker로 판정하고 결과를 반환한다.

### UC-03. custom checker로 stdio 제출 채점

사용자는 C++ stdio 코드를 제출한다. 시스템은 테스트케이스별 stdin으로 실행한 뒤 `checkerId`에 연결된 JavaScript checker로 판정하고 결과를 반환한다.

### UC-04. 컴파일 실패 처리

사용자는 잘못된 C++ 코드를 제출한다. 시스템은 컴파일 에러를 반환하고 실행 단계는 수행하지 않는다.

### UC-05. 시간 제한 초과 처리

사용자의 코드가 제한 시간을 초과한다. 시스템은 실행을 종료하고 TLE를 반환한다.

### UC-06. 메모리 제한 초과 처리

사용자의 코드가 허용된 메모리 상한을 넘는다. 시스템은 실행을 종료하거나 메모리 한계 도달을 감지하고 MLE를 반환한다.

### UC-07. 출력 제한 초과 처리

사용자의 코드가 과도한 stdout 또는 stderr를 발생시킨다. 시스템은 출력 truncate 없이 OLE를 반환한다.
